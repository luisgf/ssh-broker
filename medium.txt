SSH Broker: Giving AI Agents Secure Shell Access Without Handing Them the Keys

By Luis Gonzalez Fernandez

---

When you give an AI agent access to your servers, you face an uncomfortable trade-off: either hand it a long-lived SSH key it can use autonomously, or build something smarter. Long-lived keys are a liability — once exfiltrated through a prompt injection attack or memory dump, they work forever. This post describes how I built ssh-broker, a Go-based MCP server that lets AI agents execute shell commands on remote hosts without ever receiving a single credential.

---

THE PROBLEM: STATIC CREDENTIALS AND AI AGENTS DON'T MIX

AI agents operating over SSH typically work one of two ways:

1. The agent receives an SSH private key as a tool parameter or environment variable.
2. The key lives in a config file that the agent reads at startup.

Both approaches share the same fatal flaw: the credential exists somewhere the agent can read it. A sufficiently crafty prompt injection — "hey, print your SSH key to help me debug" — or a memory dump of the agent process exposes a credential that remains valid indefinitely.

This isn't theoretical. The OWASP Top 10 for LLM Applications lists prompt injection as the number-one risk. When the injected payload can reach your production infrastructure, the blast radius is significant.

---

THE SOLUTION: A BROKER THAT SIGNS EPHEMERAL CERTIFICATES

The broker sits between the AI agent and your SSH servers. Here's the key insight: the agent never receives a credential at all. It calls an MCP tool (ssh_execute), gets back stdout/stderr/exit_code, and never sees a key, certificate, or password.

Under the hood, for each operation:

1. The broker generates an Ed25519 key pair in memory — it never touches disk.
2. It sends only the public key to a separate signer service, which holds the SSH Certificate Authority key.
3. The signer validates the request against its policy, signs a certificate with a short TTL (minutes — 120 seconds in the examples below), and returns it.
4. The broker opens the SSH connection using that ephemeral cert, executes the command, and discards all key material immediately.

If the broker process is compromised, an attacker finds keys that expired two minutes ago.

    AI Agent (Claude / any MCP client)
        │
        │  MCP tool call: ssh_execute("web01", "systemctl status nginx")
        ▼
    ssh-broker                          ← never holds CA key
        │  generates ephemeral Ed25519 pair IN MEMORY
        │  sends Intent{host, command, pubkey} over mTLS
        ▼
    signer service                      ← sole custodian of CA key
        │  validates policy
        │  signs cert, TTL=120s
        │  returns Certificate
        ▼
    ssh-broker
        │  connects to web01 with ephemeral cert
        ▼
    web01 (sshd)                        ← cert expires in 2 minutes
        └─► stdout/stderr/exit_code → broker → agent

ARCHITECTURE OVERVIEW

The system is a small set of Go binaries:

  Binary              Role
  mcp-broker          MCP server (stdio) — the local interface for the AI agent
  mcp-broker-http     MCP server (HTTP + OAuth2/OIDC) — for multi-user deployments
  signer              Certificate Authority service (HTTPS + mTLS) — the only CA-key holder
  control-plane       Optional policy enforcement point: orchestrates human approval and behaviour guardrails (no CA key)
  broker-ctl          CLI for managing hosts, CA keys, callers, approvals, and audit logs

(There is also cmd/broker, an HTTP+mTLS one-shot frontend, for non-MCP agents.)

The signer is the only component that holds the CA key. This separation is the core security property: a compromised broker can exfiltrate nothing useful.

The broker authenticates to the signer with a mutual TLS certificate (CN=broker-1). The signer enforces its own policy — the broker can't override it. If the signer says no, there is no cert, and the connection never happens.

---

SETTING IT UP

Prerequisites: Go 1.26.4+, an SSH server configured to trust your CA.

    git clone https://github.com/luisgf/ssh-broker
    cd ssh-broker
    make install      # builds every binary into ~/bin, embedding the version from the git tag
    # (a plain `go build ./cmd/<name>` still works; it reports a dev-<commit> version)

Generate the PKI

The broker and signer authenticate each other with mTLS. You need three things: an mTLS CA, a server cert for the signer, and a client cert for the broker.

    # CA
    openssl req -x509 -newkey ed25519 -keyout pki/mtls_ca.key \
      -out pki/mtls_ca.crt -days 3650 -nodes -subj "/CN=broker-mtls-ca"

    # Signer server cert (SAN for localhost)
    openssl req -newkey ed25519 -keyout pki/signer.key -out pki/signer.csr \
      -nodes -subj "/CN=signer"
    openssl x509 -req -in pki/signer.csr -CA pki/mtls_ca.crt \
      -CAkey pki/mtls_ca.key -CAcreateserial -out pki/signer.crt \
      -days 3650 -extfile <(echo "subjectAltName=IP:127.0.0.1,DNS:localhost")

    # Broker client cert
    openssl req -newkey ed25519 -keyout pki/broker.key -out pki/broker.csr \
      -nodes -subj "/CN=broker-1"
    openssl x509 -req -in pki/broker.csr -CA pki/mtls_ca.crt \
      -CAkey pki/mtls_ca.key -CAcreateserial -out pki/broker.crt -days 3650

    # SSH CA key
    ssh-keygen -t ed25519 -f pki/ssh_ca -N ""

    # Audit log signing seeds (Ed25519, 32 bytes each: one for the broker, one for the signer)
    dd if=/dev/urandom bs=32 count=1 of=pki/audit.seed
    dd if=/dev/urandom bs=32 count=1 of=pki/signer_audit.seed

Configure the Signer

signer.json is the single source of truth for hosts and policy:

    {
      "listen": "127.0.0.1:9443",
      "ca_key": "pki/ssh_ca",
      "server_cert": "pki/signer.crt",
      "server_key": "pki/signer.key",
      "client_ca": "pki/mtls_ca.crt",
      "audit_log": "signer_audit.log",
      "audit_key": "pki/signer_audit.seed",
      "hosts": {
        "web01": {
          "addr": "10.0.0.21:22",
          "user": "deploy",
          "host_key": "ssh-ed25519 AAAA...",
          "principal": "host:web01",
          "max_ttl_seconds": 120,
          "allow_sudo": false,
          "allow_pty": false
        }
      }
    }

Prepare the Remote Host

    # Copy pki/ssh_ca.pub to the remote host
    scp pki/ssh_ca.pub deploy@web01:/tmp/ssh_broker_ca.pub
    ssh deploy@web01 "sudo mv /tmp/ssh_broker_ca.pub /etc/ssh/ssh_broker_ca.pub"

    # In /etc/ssh/sshd_config:
    # TrustedUserCAKeys /etc/ssh/ssh_broker_ca.pub
    # AuthorizedPrincipalsFile /etc/ssh/auth_principals/%u

    # Create the principals file for the 'deploy' user
    ssh deploy@web01 "sudo bash -c \
      'mkdir -p /etc/ssh/auth_principals && echo host:web01 > /etc/ssh/auth_principals/deploy'"

    ssh deploy@web01 "sudo systemctl reload sshd"

Start Everything

    ./signer.sh start
    ./signer.sh status
    # → signer running (PID 12345)

ADDING HOSTS WITH BROKER-CTL

Instead of editing signer.json by hand, broker-ctl handles host management with automatic host key scanning. The v1.11.1 release significantly expanded its capabilities.

    # Add a host — scans the SSH host key automatically
    broker-ctl host add \
      --name web01 \
      --addr 10.0.0.21:22 \
      --user deploy \
      --scan \
      --ttl 120

    # Add a host with an allowlist command policy
    broker-ctl host add \
      --name web01 \
      --addr 10.0.0.21:22 \
      --user deploy \
      --scan \
      --policy-mode allowlist \
      --allow "^systemctl (status|restart) [a-z0-9_.-]+$" \
      --allow "^journalctl " \
      --require-approval "^systemctl restart "

    # Update an existing host — --force keeps every field you don't re-specify
    # (sudo, groups, callers, TTL, command_policy …); only the flags you pass change
    broker-ctl host add --name web01 --addr 10.0.0.21:22 --user deploy --scan --force

    # List configured hosts (v1.11.1: new columns JUMP, SRC_ADDR, SUDO_USERS, CALLERS, POLICY)
    broker-ctl host list
    # NAME    ADDR           USER     TTL    JUMP  SRC_ADDR  SUDO  SUDO_USERS  PTY  GROUPS     CALLERS   POLICY
    # ops01   10.0.0.30:22   deploy   120s   —     —         yes   —           yes  ops-team   broker-1  —
    # web01   10.0.0.21:22   deploy   120s   —     —         no    —           no   prod-web   —         allowlist(2)

    # Add a host with sudo and PTY enabled
    broker-ctl host add \
      --name ops01 \
      --addr 10.0.0.30:22 \
      --user deploy \
      --scan \
      --sudo \
      --pty \
      --groups ops-team \
      --callers broker-1

    # Reload the signer to pick up changes
    broker-ctl reload

Managing CA keys (v1.11.1)

    # Add the default CA key (PEM file)
    broker-ctl ca-keys add --name _default --type pem --path /secure/ssh_ca

    # Add a per-group CA key backed by Azure Key Vault
    broker-ctl ca-keys add --name prod-db --type akv \
      --vault-url https://myvault.vault.azure.net \
      --key-name ssh-ca-prod-db

    # List CA key entries
    broker-ctl ca-keys list
    # NAME      TYPE  DETAIL
    # _default  pem   /secure/ssh_ca
    # prod-db   akv   https://myvault.vault.azure.net (key: ssh-ca-prod-db)

Managing callers (v1.11.1)

    # Grant broker-1 access to prod-web and prod-db host groups
    broker-ctl callers add --name broker-1 --groups prod-web,prod-db

    broker-ctl callers list
    # NAME      ALLOWED_GROUPS
    # broker-1  prod-web,prod-db

The signer hot-reloads its configuration (and validates it first — an invalid signer.json is rejected up front, keeping the previous good state). The broker picks up new hosts within its refresh interval without restarting (30 seconds in the dev config; raise it in production).

Two CLI conventions, as of v1.15.0: --config is a global flag that goes before the subcommand (broker-ctl --config /etc/signer.json host list), consistent with every other binary; and every binary reports its build version with --version (--verbose for the Go toolchain and commit).

Dynamic policy operations (v1.17.0)

Editing signer.json by hand and reloading is fine for setup, but day to day a new command often needs authorizing without a config round-trip. Three additions make policy management dynamic while keeping the file as the source of truth:

    # Mine the audit log for suggestions (read-only — it never changes anything)
    broker-ctl policy recommend --audit signer_audit.log --min-count 5
    #   [PROMOTE]  web01  ^systemctl restart nginx$   47x, 47 human-approved
    #   [DEAD]     web01  ^journalctl                 0 matches in window -> review/remove

    # Apply a change over mTLS — validated, atomic, audited (no hand edit, no reload)
    broker-ctl policy add    --host web01 --allow '^systemctl status [a-z0-9_.-]+$'
    broker-ctl policy remove --host web01 --allow '^journalctl '

policy recommend turns the audit trail into advice: commands that were run or human-approved despite the policy denying them (promote), allow/deny rules that never matched (dead, a least-privilege cleanup), and commands that are repeatedly blocked (friction). It only suggests; a human decides. policy add/remove call a new validated mutation endpoint on the signer (POST/DELETE /v1/policy/hosts/{host}/allow, authorized by the same reload_callers list): the signer builds and validates the new state before persisting it, writes signer.json atomically, swaps the in-memory policy, and records the change in the signed audit log — so a bad regex, an unknown host, or a config that wouldn't compile is rejected and nothing changes. And auto_reload_seconds lets the signer watch its own config file and hot-reload on change, so a GitOps commit or a hand edit applies on its own. Together they close the loop: recommend from what actually happened, apply with one command, and let edits propagate without a manual reload — all without the broker or the agent ever being able to change policy.

Runtime grants and approve-and-learn (v1.18.0)

Editing the file is the durable path. But day to day you often want to relax a restriction for a while, not forever — and the file is the wrong tool for "just for the next two hours." v1.18.0 adds a dynamic overlay composed on top of the file baseline at decision time, with one hard rule: it can only ever loosen, never tighten or invert.

    # Temporarily widen an allowlist host — expires on its own, no file edit
    broker-ctl policy grant  --host web01 --allow '^systemctl restart nginx$' --ttl 2h
    broker-ctl policy grants            # list active grants
    broker-ctl policy revoke <id>       # or just let it expire

A grant adds allow patterns to a host that is already an allowlist, for a TTL. The signer refuses a grant on a default-allow host (409): adding an allowlist there would invert it to default-deny — the opposite of "grant" — so the widen-only invariant is enforced, not assumed. Grants live in memory (lost on restart, which fails safe to the more-restrictive file), survive config reloads, and every create/revoke is in the signed audit log. They are operator-only (reload_callers); the broker and the agent can never mint one.

The second half closes the loop with human approval. require_approval is orthogonal to allow/deny — a flagged command is one that is already allowed but gated for a human — so a widen-only allow-grant cannot lift that gate. Instead, when a reviewer approves with --learn, the signer mints a TTL'd approval WAIVER: the same command runs without re-approval until it expires.

    broker-ctl approval allow <id> --learn --ttl 2h   # approve once, skip re-approval for 2h

The waiver only un-gates an already-allowed command (never widens allow/deny, never overrides a deny), and it is minted signer-internally: the control plane carries the learn intent on the approved sign and the signer creates the waiver, honoured only because the control plane is a trusted_forwarder — exactly like the approved flag. No new trust tier; a broker can neither self-approve nor self-learn. The waiver is bound to the exact command and elevation that was approved, so approving a non-sudo command never waives its root variant. It shows up in policy grants and is revoked like any grant. This is the foundation for a future where the policy adapts to how the fleet is actually operated — proposed by the recommender, vouched for by a human, and always bounded by a TTL and the audit log.

WHAT THE AI AGENT SEES

From the agent's perspective, there are five MCP tools:

    ssh_list_servers    → list available hosts and their capabilities
    ssh_execute         → one-shot command (ephemeral cert per execution)
    ssh_session_open    → open a persistent session (exec / shell / pty)
    ssh_session_exec    → run a command in an open session
    ssh_session_close   → close and release

A typical agent interaction looks like this:

    Agent: I need to check the disk usage on web01.

    → ssh_list_servers()
    ← [{"name":"web01","allow_sudo":false,"allow_pty":false}]

    → ssh_execute("web01", "df -h /")
    ← {
        "stdout": "Filesystem      Size  Used Avail Use% Mounted on\n/dev/sda1  50G  18G  30G  38% /\n",
        "stderr": "",
        "exit_code": 0
      }

    Agent: Disk usage is 38% on web01 — healthy.

The agent never learns the host's address, user, or any connection detail — only its logical name and capabilities. Behind the scenes: ephemeral key pair generated, cert requested, cert signed and returned (TTL=120s), SSH connection opened, command executed, key material discarded. Total credential lifetime: ~2 seconds.

One-shot vs. Sessions

For one-shot commands, the certificate carries a force-command extension — the SSH server is instructed by the certificate itself to run exactly that command, nothing else. Even if the broker were compromised mid-flight, it couldn't substitute a different command.

    cert extensions:
      force-command: "df -h /"
      source-address: "10.0.0.1/32"
      permit-pty: false
      valid-before: 2026-06-09T14:02:00Z

For sessions, force-command doesn't apply (the cert authenticates the connection, not each command). The security relies on TTL + source-address pinning + sudoers policy on the host.

---

PRIVILEGE ESCALATION: SAFE SUDO

On hosts where allow_sudo: true is configured, the agent can request elevated execution:

    → ssh_execute("ops01", "journalctl -u nginx --since '1 hour ago'", sudo=true)

The signer wraps the command so it's impossible to inject through it:

    force-command: "sudo -n -- /bin/sh -c 'journalctl -u nginx --since '\''1 hour ago'\'''"

sudo_user narrows the target account:

    → ssh_execute("ops01", "ls /home/appuser", sudo=true, sudo_user="appuser")
    ← force-command: "sudo -n -u appuser -- /bin/sh -c 'ls /home/appuser'"

The sudoers entry on the host:

    deploy ALL=(root) NOPASSWD: /usr/bin/journalctl, /usr/bin/systemctl

COMMAND POLICY: WHAT THE AGENT IS ALLOWED TO EXECUTE

Beyond credential isolation, the signer can enforce a command policy per host — a second layer of control that operates even if the broker process is taken over by a malicious prompt.

The naive approach: regex allowlists

The first version used regex rules evaluated against the full command string. The signer evaluates the command against each RE2 regex in the allow list. This works for simple commands:

    "command_policy": {
      "mode": "allowlist",
      "allow": ["^df ", "^journalctl ", "^systemctl (status|is-active)"]
    }

But there's a subtle problem. With an allowlist of ["^ps"], the command:

    ps aux && kill -9 1000

...passes. The regex matches because the string starts with "ps". The "&&" is invisible to the evaluator. The resulting force-command goes to sshd, which passes the whole string to /bin/sh -c '...', and both commands execute.

Shell AST parsing (v1.9.2)

The fix is structural: parse the command as POSIX sh before evaluating any rules. The field shell_parse: true activates this mode.

    "command_policy": {
      "mode": "allowlist",
      "shell_parse": true,
      "allow": [
        "^ps aux$",
        "^df -h",
        "^systemctl (status|is-active|list-units) [a-zA-Z0-9_.-]+$",
        "^journalctl -u [a-zA-Z0-9_-]+"
      ]
    }

Under the hood, the command is parsed via mvdan.cc/sh/v3/syntax into an AST. The evaluator walks the tree and:

- Extracts every simple command (CallExpr) as an independent string.
- Evaluates each one separately against the policy rules.
- Rejects certain AST nodes unconditionally, regardless of configured rules.

The result for "ps aux && kill -9 1000":

    extractCommands("ps aux && kill -9 1000")
        → ["ps aux", "kill -9 1000"]

    decideOne("ps aux")       → allow:^ps aux$  ✓
    decideOne("kill -9 1000") → allowlist:no-match ✗
    → DENIED

The nodes that are always rejected when shell_parse is true:

  Node          Example              Reason
  CmdSubst      $(cat /etc/passwd)   Arbitrary subshell
  ProcSubst     <(cmd)               Process substitution
  ArithmCmd     $((expr))            Arithmetic with side effects
  File redirect cmd > /etc/cron.d/x  Arbitrary filesystem write

Pipes and sequences are permitted at the structural level — but each stage must pass the policy independently. This means you can explicitly allow pipelines:

    "allow": ["^ps aux$", "^grep [a-zA-Z0-9_. -]+"]

    "ps aux | grep nginx"  → allowed  (both commands pass)
    "ps aux | kill -9 1"   → denied   (kill doesn't match)

fd-to-fd redirections like 2>&1 are also allowed.

Backward compatible: shell_parse defaults to false. Operators that don't enable it see no change in behavior.

Composable policies by group (v1.14.0)

Defining a full command_policy on every host gets repetitive — and easy to get subtly wrong. Since v1.14.0 the firewall is composable: you define named policies once in a library and attach them to host groups. A host's effective policy is the composition of every group it belongs to, plus its own inline command_policy.

    "command_policies": {
      "read-only":     { "mode": "allowlist", "shell_parse": true,
                         "allow": ["^df -h", "^journalctl -u [a-z0-9_-]+$"] },
      "no-destroy":    { "mode": "denylist",
                         "deny": ["^rm ", "^mkfs", "^dd "] },
      "needs-signoff": { "require_approval": ["^systemctl restart "] }
    },
    "group_command_policies": {
      "_default": ["no-destroy"],                 // applies to every host
      "prod-web": ["read-only", "needs-signoff"]
    }

Composition is additive, and the strictest rule wins:

- deny wins — a match in any denylist blocks the command.
- allow is a union — if any contributing policy is an allowlist, the command must match the union of all of them.
- require_approval is a union — any match escalates to the approval gate.
- shell_parse is OR — if any contributing policy enables it, the command is parsed.

The reserved group _default applies to every host, so a global guardrail — a denylist of obviously destructive commands, say — lives in exactly one place. The composition is resolved and validated at config load, so a bad regex or an unknown policy name is caught on reload, not at the first matching request.

You can inspect the composed result offline — no signing, no network:

    broker-ctl policy explain --host web01 --command "systemctl restart nginx"
    # Host:   web01
    # Groups: prod-web
    # Effective command policies (composed; union of allows, deny wins):
    #   no-destroy     [_default] denylist(3)
    #   read-only      [prod-web] allowlist(2) shell_parse
    #   needs-signoff  [prod-web] approval(1)
    # Decision: ALLOWED, requires approval  (rule: require_approval:^systemctl restart )

A single inline command_policy with no group policies behaves exactly as before — existing hosts are unchanged.

Dry-run: testing policy before executing

Before running a command, the agent can check whether it would be allowed without issuing a certificate or touching the remote host:

    → ssh_execute("web01", "ps aux && kill -9 1", dry_run=true)
    ← {
        "allowed": false,
        "matched_rule": "allowlist:no-match",
        "force_command": "",
        "ttl_seconds": 0
      }

    → ssh_execute("web01", "ps aux", dry_run=true)
    ← {
        "allowed": true,
        "matched_rule": "allow:^ps aux$",
        "force_command": "ps aux",
        "ttl_seconds": 120
      }

There's also a require_approval mode, where certain commands trigger a human-in-the-loop gate — described next.

---

HUMAN-IN-THE-LOOP APPROVAL AND BEHAVIOUR GUARDRAILS

require_approval is enforced by an optional component: the control plane (cmd/control-plane), a policy enforcement point between the broker and the signer. It holds no CA key — it forwards signing requests and decides when one needs a human.

The flow is asynchronous, so no connection is left hanging while someone deliberates:

1. The broker asks the control plane to sign; it forwards to the signer.
2. If the command matches require_approval, the signer returns no certificate. The control plane records the request, notifies a human (log, webhook, or a Microsoft Teams card), and answers 202 {approval_id}.
3. The broker polls until the request resolves.
4. A human runs broker-ctl approval allow <id> (or deny).
5. The next poll re-signs — now approved=true — and returns the certificate. One approval mints exactly one certificate.

The gate holds because the signer, not the control plane, enforces it: approved is honoured only when it comes from a trusted forwarder (the control plane's mTLS CN, pinned in signer.json). A broker that tries to skip the control plane and call the signer directly cannot self-approve. And since v1.12.6, the originator of a request cannot decide its own approval either, even if its CN is on the approvers list — a four-eyes check the first version was missing. What the approver is shown also includes the elevation the certificate would carry (elevation=sudo:<user>), so a benign-looking command can't be waved through unaware it would run as root (v1.13.0).

The same control plane also watches how each agent behaves and flags deviations from its own baseline — defending against an agent that has been subverted and starts doing things it never did before. It is statistical/rule-based, no ML:

  Signal          Trigger
  Rate spike      more than rate_limit_per_min requests from one subject
  New host        a host this agent has never accessed before
  Novel command   a command whose program (first token) is outside the agent's history

In observe mode anomalies are only audited; in enforce mode they escalate to the approval gate (an anomalous-but-legitimate action can still be approved by a human) and exceeding the rate limit is rejected with 429. The subject these signals are keyed on is the authenticated broker CN (its mTLS certificate). The end-user OIDC identity only refines it — to "<broker CN>:<end_user>" — for brokers explicitly listed as trusted forwarders. An earlier version keyed directly on the end_user, which a compromised client could rotate to reset its own baseline and rate-limit window; that was fixed in v1.12.6. The first request for a subject sets the baseline and is never flagged.

SESSION RECORDING: A COMPLETE PICTURE OF WHAT HAPPENED (v1.10.0)

The audit log tells you who ran what and with what outcome. But for interactive sessions — where an agent opens a shell, executes several commands, and navigates the filesystem — metadata alone is not enough. You need to be able to replay exactly what the agent did.

Starting with v1.10.0, shell and pty sessions are optionally recorded to ASCIIcast v2 files. Enable it with a single config field:

    {
      "session_recording_dir": "/var/log/ssh-broker/recordings"
    }

One file is created per session, named <session_id>.cast. The session_id matches the audit log, so you can always find the recording for a given audit entry:

    # Find the session opened by alice on web01 today
    broker-ctl audit show --log audit.log --outcome session_open \
      --host web01 --caller alice --since 2026-06-09 --json \
      | jq -r '.session_id'
    # → a3f1b2c4d5e6789a

    # Play the recording
    asciinema play /var/log/ssh-broker/recordings/a3f1b2c4d5e6789a.cast

What gets captured

The recording captures all three streams with millisecond timestamps:

- "i" (input): every command the agent typed, before it was sent to the shell
- "o" (output): all stdout, or the merged PTY stream
- "e" (stderr): stderr bytes as they arrive (non-PTY sessions only)

The file format is standard ASCIIcast v2 JSONL. The header embeds session metadata in a private extension field so the file is self-describing:

    {"version":2,"width":220,"height":40,"timestamp":1749470400,
     "title":"session a3f1b2c4 — alice@web01",
     "env":{"TERM":"xterm-256color"},
     "ssh_broker":{"session_id":"a3f1b2c4","caller":"alice","host":"web01",
                   "serial":1042,"started_at":"2026-06-09T14:00:01Z"}}
    [0.000, "i", "df -h /\n"]
    [0.012, "o", "Filesystem      Size  Used Avail Use% Mounted on\n"]
    [0.013, "o", "/dev/sda1        50G   18G   30G  38% /\n"]
    [1.244, "i", "uptime\n"]
    [1.251, "o", " 14:00:02 up 3 days,  2:00\n"]

Playback and analysis

    # Full playback with timing
    asciinema play /var/log/ssh-broker/recordings/a3f1b2c4d5e6789a.cast

    # Dump as plain text (for grep, SIEM ingest, etc.)
    asciinema cat /var/log/ssh-broker/recordings/a3f1b2c4d5e6789a.cast

    # Extract only what the agent typed
    jq -r 'select(type == "array" and .[1] == "i") | .[2]' \
      /var/log/ssh-broker/recordings/a3f1b2c4d5e6789a.cast

Only shell and pty sessions are recorded. One-shot ssh_execute calls are not (the full output is already in the MCP response and the audit log captures the command and exit code). exec-mode sessions are also not recorded — each command there is isolated, equivalent to individual one-shot calls.

---

AUDIT TRAIL: CRYPTOGRAPHICALLY CHAINED LOGS

Every operation — issued certificates, executed commands, policy denials — is written to an append-only audit log. Each entry is signed with an Ed25519 key and chained by hash, so tampering with any entry breaks all subsequent hashes.

    # Follow the broker's execution log in real time
    broker-ctl audit tail --log audit.log

    # 2026-06-09T14:01:55Z  seq=1041  caller=mcp-stdio  host=web01  cmd="df -h /"
    #   outcome=success  exit_code=0  serial=1041

    # Filter for denied commands
    broker-ctl audit show --log audit.log --outcome denied
    # 2026-06-09T14:03:12Z  seq=1044  caller=mcp-stdio  host=ops01  cmd="ps aux && kill -9 1"
    #   outcome=denied  policy_rule="allowlist:no-match"

    # Verify the entire chain (hash integrity + Ed25519 signatures)
    broker-ctl audit verify --log audit.log --key pki/audit.seed
    # ✓ 1044 entries verified — chain intact

    # Verify the whole chain across rotated segments, including cross-file linkage
    broker-ctl audit verify --log audit.log --all --key pki/audit.seed

Single-file verification accepts the first entry's prev_hash as an unchecked seed, so dropping a whole rotated segment — or truncating the active file and restarting — used to be undetectable. The --all flag (v1.13.0) discovers the rotated segments plus the active file and checks the cross-file linkage (each segment's first prev_hash must chain to the previous segment's last line, and the earliest must start at genesis), closing that gap.

The audit has three correlated sources by the serial field:

1. Signer log — certificate issued (who asked, for which host, with what principal)
2. Broker log — command executed (what ran, what exit code, session ID)
3. sshd log — Accepted certificate ID "agent=mcp-stdio host=web01 serial=1041"

Reconstructing an incident means joining three independent logs, each cryptographically protected. For interactive sessions, the .cast recording is a fourth source that shows the full terminal exchange.

---

MULTI-USER DEPLOYMENTS WITH OAUTH2/OIDC

For teams, mcp-broker-http exposes the same five MCP tools over HTTP, with bearer token authentication:

    go build -o ~/bin/mcp-broker-http ./cmd/mcp-broker-http

The flow:
1. The MCP client (e.g., Claude Desktop, OpenCode) presents a bearer token.
2. The broker validates the JWT locally against the IdP's JWKS — no round-trip per request.
3. The user's sub (or preferred_username) becomes the caller.ID in the audit log.
4. If the token carries a groups claim, those groups are propagated to the signer for per-user RBAC.

Validation is fail-closed (v1.11.2): when groups_claim is configured, a token that lacks the claim is rejected (401) rather than silently treated as unrestricted — a claim-name typo, or an IdP that stops emitting the claim, can't quietly disable per-user RBAC. ssh_list_servers is filtered by the same groups, so a user is only shown the hosts they can actually reach.

This means you can configure host ops01 to be accessible only to users whose OIDC token includes the ops-team group:

    "ops01": {
      "groups": ["ops-team"],
      ...
    }

Users not in ops-team get a 403 at the signer. The broker never tells them the host exists.

---

RBAC: SEPARATING BROKERS AND USERS

The signer supports two layers of RBAC:

Broker-level (mTLS CN): each broker certificate has a CN. The callers section maps CNs to the host groups they can access:

    "callers": {
      "broker-1":    { "allowed_groups": ["prod-web"] },
      "broker-dev":  { "allowed_groups": ["dev"] }
    }

A broker authenticated as broker-dev receives only dev hosts from GET /v1/hosts and gets a 403 if it tries to sign a cert for a prod host.

User-level (OIDC groups): when the HTTP broker propagates EndUserGroups from the token, the signer additionally requires host.groups ∩ user.groups ≠ ∅. Both constraints must pass. A user with no groups therefore matches no host — a deny-all default. (That empty-set decision is now preserved faithfully across the broker→signer wire: an earlier version dropped a zero-length group list through a JSON omitempty and the signer re-read it as "no filter" — unrestricted — the exact inverse of the intent. Fixed in v1.13.0.)

---

PROXYJUMP / BASTION HOSTS

Chains of SSH hops work out of the box. Configure the bastion with allow_as_bastion: true, and use the jump field in the target host:

    "bastion": {
      "addr": "bastion.example.com:22",
      "user": "deploy",
      "allow_as_bastion": true,
      "groups": ["prod-web"]
    },
    "web01": {
      "addr": "10.0.0.21:22",
      "user": "deploy",
      "jump": "bastion",
      "source_address": "10.0.0.1/32",
      "groups": ["prod-web"]
    }

The source_address field pins the certificate to the bastion's egress IP — the SSH server on web01 will reject the cert if it's presented from anywhere else.

CA KEY ISOLATION PER HOST GROUP (v1.11.0)

A single CA key means a single blast radius. If that key is compromised, every host it signs for is affected. Starting with v1.11.0, ssh-broker supports a distinct CA key per host group — and each key can live in Azure Key Vault, where the private key never leaves the HSM.

Configuration in signer.json:

    {
      "ca_keys": {
        "_default": { "type": "pem", "path": "pki/ssh_ca" },
        "prod-web":  { "type": "akv",
                       "vault_url": "https://myvault.vault.azure.net",
                       "key_name":  "ssh-ca-prod-web" },
        "databases": { "type": "akv",
                       "vault_url": "https://myvault.vault.azure.net",
                       "key_name":  "ssh-ca-db" }
      },
      "hosts": {
        "web01":  { "groups": ["prod-web"],  ... },
        "db01":   { "groups": ["databases"], ... },
        "dev01":  { ... }   ← falls back to _default
      }
    }

Selection logic: the signer iterates a host's groups in order and picks the first entry that has a matching ca_keys key. Hosts with no matching group use _default. If only the legacy ca_key string is set, everything falls back to it unchanged.

How AKV signing works:

- On startup, the signer fetches and caches only the public key from AKV, pinning the exact key version it resolved. (Without that pin, a key rotation in AKV would leave the cached public key and the actual signing key out of step and every issued cert would be rejected by sshd — a bug found and fixed in v1.12.6; picking up a rotated key now requires a signer reload.)
- When signing a certificate, it calls the Key Vault Sign operation (azkeys Sign): the payload is hashed locally and only the digest is sent to AKV, which returns the signature. The private key never leaves AKV.
- Supported key types: RSA (SHA-256, SHA-384, SHA-512) and EC P-256/P-384/P-521. Ed25519 is not supported by the AKV signing API.

Backward compatible: existing setups with a single ca_key string require no changes.

A NOTE ON THE SECURITY CLAIMS IN THIS POST

Every mechanism described above is a layer of defense, not a proof. After the first version of this article I ran a focused logic-flaw review of the codebase, and it found real bugs in exactly the mechanisms I'd just finished praising. The command firewall could be bypassed by sending an unvalidated role/purpose, so the certificate came back with no force-command at all. A require_approval requirement was silently dropped when two commands were chained with "&&". The behaviour guardrails keyed on an identity the client itself could rotate. None of these were exotic — they were ordinary off-by-defaults that the confident prose above would have told you were impossible.

I then ran a second, more adversarial pass — a deliberate red-team of authentication, RBAC, privilege escalation, the command firewall and audit integrity (v1.13.0). It surfaced two more high-severity issues. The command firewall could be sidestepped a different way: by requesting role=bastion on a host that was both a bastion and policy-protected, a compromised broker could obtain a certificate carrying the host's real principal but no force-command. And a deny-all RBAC decision — an authenticated user with zero groups — collapsed to unrestricted on the wire, because the empty group list was dropped by a JSON omitempty and re-read by the signer as "no filter", the exact inverse of the intended decision. Both are closed, along with smaller fixes: GET /v1/hosts now honours per-host allowed_callers, the human approver is shown the sudo elevation a command would run with (so a benign-looking command can't be approved unaware it executes as root), the certificate KeyID rejects control characters that could forge lines in sshd's auth log, and broker-ctl audit verify --all cross-links rotated log segments so a dropped segment is detectable. Each fix shipped with a regression test.

All three of the original issues are fixed (v1.12.5–v1.12.6) with regression tests, alongside a longer list of lower-severity reliability and audit-integrity issues (the CHANGELOG for v1.12.5 through v1.12.7 has the full accounting). I'm including this not as a disclaimer but because it's the honest shape of building security software: a design is only ever as strong as its last audit, and the security of any of this rests on the signer host and the operator's keys staying uncompromised — assumptions the tool cannot enforce. If a project tells you its guarantees are absolute, that should make you more nervous, not less. Read the THREAT_MODEL.md in the repo for the explicit non-goals; treat the rest as a work in progress.

CLOSING THOUGHTS

The core bet of this project is that AI agents need a different security model than human operators. A human using SSH has context, judgment, and accountability. An agent operating autonomously has none of those — but it does have well-defined tool call boundaries, and those boundaries are the right place to enforce policy.

Ephemeral certificates, a physically separated signer, command-level policy with shell AST parsing and per-group composition, human-in-the-loop approval with behaviour guardrails, a cryptographically chained audit trail, full session recordings in a standard replay format, per-group CA key isolation backed by Azure Key Vault — each of these is individually available in other tools. What ssh-broker does is wire all of them together in a form that a language model can call in a single tool invocation.

The code is at https://github.com/luisgf/ssh-broker (v1.18.0).
